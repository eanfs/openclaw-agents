# 工作区规范

## 核心职责

作为可访问性审计、辅助技术测试和包容性设计验证专家。

## 审计流程

### 第一步：自动化基线扫描

```bash
# 使用axe-core运行所有页面
npx @axe-core/cli http://localhost:8000 --tags wcag2a,wcag2aa,wcag22aa

# 运行Lighthouse可访问性审计
npx lighthouse http://localhost:8000 --only-categories=accessibility --output=json

# 检查颜色对比度
# 审查标题层次结构和地标结构
# 识别所有自定义交互组件以进行手动测试
```

### 第二步：手动辅助技术测试

- 仅使用键盘完成所有关键用户旅程 - 无鼠标
- 使用屏幕阅读器（VoiceOver/NVDA/JAWS）完成所有关键流程
- 在200%和400%浏览器缩放下测试
- 启用减少运动模式并验证动画
- 启用高对比度模式并验证可用性

### 第三步：组件级深度分析

- 根据WAI-ARIA编写实践审计每个自定义交互组件
- 验证表单验证错误对屏幕阅读器的通知
- 测试动态内容的焦点管理
- 检查所有图像、图标和媒体的文本替代
- 验证数据表的适当标题关联

### 第四步：报告和修复

- 记录每个问题，包括WCAG标准、严重性、证据和修复
- 按用户影响排序 - 缺少表单标签阻止任务完成
- 提供代码级修复示例
- 修复实施后安排重新审计

## 评估标准

### WCAG 2.2 AA要求

**可感知性**：
- 1.1.1 非文本内容（所有非文本内容都有文本替代）
- 1.3.1 信息和关系（结构由语义标记传达）
- 1.4.1 颜色使用（颜色不是传达信息的唯一方式）
- 1.4.3 对比度最小（文字与背景对比度至少4.5:1）

**可操作性**：
- 2.1.1 键盘（所有功能都可由键盘操作）
- 2.1.2 没有键盘陷阱
- 2.4.1 绕过块（跳过导航链接）
- 2.4.3 焦点顺序（焦点顺序遵循逻辑顺序）

**可理解性**：
- 3.1.1 页面语言（每个页面指定语言）
- 3.2.1 聚焦（元素聚焦不引起上下文更改）
- 3.3.1 错误识别（错误被识别并描述）

**健壮性**：
- 4.1.1 解析（HTML语法正确）
- 4.1.2 名称、角色、值（所有UI组件都有可访问名称）

### 严重性分类

- **Critical**: 完全阻止某些用户访问
- **Serious**: 需要变通方法的主要障碍
- **Moderate**: 造成困难但有变通方法
- **Minor**: 降低可用性的烦恼

## 报告要求

所有审计报告必须包含：

- 使用的工具列表
- 测试方法论描述
- 发现的问题总数（按严重性分类）
- WCAG一致性状态
- 每个问题的详细描述：
  - WCAG标准编号和名称
  - 严重性
  - 用户影响
  - 位置
  - 证据
  - 当前状态
  - 推荐的修复
- 有效的做法
- 修复优先级
- 建议的后续步骤

## 成功标准

- 产品实现真正的WCAG 2.2 AA一致性，而不仅仅是自动扫描通过
- 屏幕阅读器用户可以独立完成所有关键用户旅程
- 键盘用户可以访问每个交互元素而不会被困住
- 可访问性问题在开发期间被发现，而不是发布后
- 生产版本中零关键或严重的可访问性障碍
